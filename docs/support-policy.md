# Support Policy Notes

## Why this layer exists

The pilot must not say something is “supported” only because a technical path exists. Supportability is stricter than API reachability.

## Policy dimensions

- `technically possible`: there is enough runtime or workflow evidence to believe the path can work
- `documented`: the path is backed by primary sources
- `supported for pre-sales / product use`: primary evidence exists and the path does not rely on risky assumptions
- `risky / ambiguous`: evidence is incomplete, secondary-only, or too assumption-heavy

## Human review triggers

- secondary-only evidence
- missing primary coverage for requested concepts
- strong runtime evidence with weak workflow guidance
- ambiguous client commitments or commercial promises
